## Known Limitations

### Non-Issues (Expected Behavior)

1. **Jarvis AI Service** - Optional, not required for core functionality
   - If not configured, AI features are disabled but app works
   - Background jobs are queued and can be processed later

2. **Test Suite** - One test requires PostgreSQL
   - This is expected and documented
   - API tests pass without database
   - Production uses real database

3. **SQLite Mode** - Limited FTS capability
   - PostgreSQL FTS is disabled in SQLite mode
   - This is documented in code
   - Production should use PostgreSQL

## Performance & Scalability

### Current Architecture

```
┌─────────────────────────────────────────────┐
│      Single Docker Compose Stack            │
├─────────────────────────────────────────────┤
│  PostgreSQL │ Redis │ MinIO │ App │ Worker │
│  (container) (container) (container) (container) (container) │
└─────────────────────────────────────────────┘
```

### Scalable Architecture (Production)

```
┌──────────┐   ┌──────────┐   ┌──────────┐
│   App    │   │   App    │   │   App    │
│ (scaled) │   │ (scaled) │   │ (scaled) │
└────┬─────┘   └────┬─────┘   └────┬─────┘
     │              │              │
     └──────────────┴──────────────┘
                    │
     ┌──────────────┼──────────────┐
     │              │              │
┌────▼────┐   ┌────▼────┐   ┌────▼────┐
│ PostgreSQL│  │  Redis  │   │  MinIO  │
│  (managed)│  │(managed)│   │  (S3)   │
└───────────┘  └─────────┘   └─────────┘
```

**To scale:**
1. Use external managed PostgreSQL (RDS, Cloud SQL, etc.)
2. Use external managed Redis (ElastiCache, Cloud Memorystore, etc.)
3. Use external S3 or managed MinIO
4. Scale app containers horizontally
5. Add load balancer
